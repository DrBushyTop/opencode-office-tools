# PowerPoint Roadmap Foundation Implementation Plan

## Overview

We will strengthen the PowerPoint add-in in four vertical phases: make Open XML mutations reliable, add first-class native business content tools, make the UI selection-aware, and then add template-aware generation primitives. The goal is to move from a capable low-level chat toolset to a dependable PowerPoint-native authoring workflow for timelines, estimates, and offer/proposal slides.

## Current State Analysis

The current PowerPoint surface is a generic task pane chat UI with a single `OpenCode` launcher button in the ribbon and no PowerPoint-specific command surface beyond that (`manifest.xml:101`). The UI does not track active slide or selection state in React; messages are sent as plain chat turns with host-specific tools enabled, but without injected PowerPoint context (`src/ui/App.tsx:254`, `src/ui/App.tsx:441`, `src/ui/lib/office-tool-bridge.ts:7`).

For document understanding, the add-in already exposes useful deck metadata: `get_presentation_structure` returns slide master/layout ids, placeholders, theme colors, selected slide ids, selected shape ids, and slide dimensions where supported (`src/ui/tools/getPresentationStructure.ts:26`). `manage_slide` can create slides with `slideMasterId` and `layoutId` (`src/ui/tools/manageSlide.ts:31`), while `add_slide_from_code` already sizes generated slides to the deck and inserts them with destination theme formatting (`src/ui/tools/addSlideFromCode.ts:117`).

The main reliability issue is the Open XML mutation path. Slide transitions, notes, and animations all rely on replacing a slide by exporting it, mutating the package, inserting the mutated slide, and deleting the original (`src/ui/tools/powerpointOpenXml.ts:1457`). Shape targeting already has an XML fallback resolver that maps a requested shape id to the current Office shape when direct lookup fails (`src/ui/tools/powerpointShapeTarget.ts:10`), but the refresh/remap behavior is not centralized enough to make multi-step sessions robust.

## Desired End State

The PowerPoint add-in should be able to:

- apply notes, transitions, and animations without leaving the agent in a stale slide/shape identity state,
- create and edit native images, tables, charts, and business diagram layouts without forcing `add_slide_from_code` for common work,
- let users target the current slide or selected shapes from the UI with clear context and shortcuts,
- generate into the current deck's template using layouts, placeholders, and theme data instead of only freeform code.

Success means a user can select one or more shapes, describe a change in the add-in UI, and have the agent execute safely against the current context; generate a project timeline or estimate table as editable PowerPoint content; and iteratively refine template-compliant slides without stale-object failures.

### Key Discoveries

- The current Open XML replacement backbone is already centralized in `replaceSlideWithMutatedOpenXml`, so the stability fix should extend that boundary rather than invent a new one (`src/ui/tools/powerpointOpenXml.ts:1457`).
- Shape-id fallback via exported XML already exists and should become the standard post-round-trip remap mechanism (`src/ui/tools/powerpointShapeTarget.ts:10`).
- Template metadata is already available in `get_presentation_structure`, but only as a text summary; it is not yet a structured generation primitive (`src/ui/tools/getPresentationStructure.ts:78`).
- The codebase already has mature patterns for adding new tools: registry JSON, UI handler module, validation, wrapper generation, and tests (`src/shared/office-tool-registry.json:688`, `src/ui/tools/manageSlideShapes.ts:1`).
- Microsoft add-ins can support ribbon commands and custom contextual tabs with task pane fallbacks, but an inline shape-anchored mini-editor is not a safe core assumption for this plan. Selection APIs such as `getSelectedSlides()` and `getSelectedShapes()` are available in PowerPointApi 1.5, while slide-selection events remain preview-only in public docs as of March 2026.

## What We're NOT Doing

- We are **not** building a full SmartArt replacement or a generic diagraming engine in this plan.
- We are **not** making inline canvas-attached prompt popovers a required experience.
- We are **not** rewriting the entire PowerPoint tool architecture or replacing the chat model.
- We are **not** attempting broad PowerPoint master/theme editing in the first pass.
- We are **not** depending on preview-only PowerPoint events for core correctness.

## Implementation Approach

Use the roadmap order as approved, but keep each phase vertically useful on its own:

1. make existing Open XML-backed features trustworthy,
2. add native business-authoring primitives users can immediately value,
3. connect those tools to selection-aware UX in the task pane and, where supported, ribbon/contextual entry points,
4. then use the reliable context + native primitives + template metadata to build template-aware generation.

This sequencing preserves validation boundaries. Each phase should be demoable before the next: first no stale ids, then native estimate/timeline content, then selection-scoped UI, then layout-aware generation.

## Phase 1: Stabilize Open XML Mutation Flows and Identity Remapping

### Overview

This phase hardens notes, transitions, animations, and any other slide-replacement path so callers can continue working after a round-trip without manually guessing fresh slide or shape ids.

### Why This Phase First

All later PowerPoint workflows depend on reliable identity handling. If transitions or animations still invalidate slide/shape references, native tools and selection-aware UI will feel random and unsafe.

### Changes Required

#### 1. Centralize round-trip result metadata

**File**: `src/ui/tools/powerpointOpenXml.ts`

**Changes**: Extend `replaceSlideWithMutatedOpenXml` so it returns structured metadata about the replacement, including old/new slide ids, final slide index, and optional shape remap helpers.

```ts
export interface OpenXmlRoundTripResult {
  originalSlideId: string;
  replacementSlideId: string;
  finalSlideIndex: number;
  shapeIdMap?: Record<string, string>;
}

export async function replaceSlideWithMutatedOpenXml(...): Promise<OpenXmlRoundTripResult> {
  // export -> mutate -> insert -> reload slides -> locate replacement -> delete original
  // return replacement metadata for downstream tools
}
```

**Why**: Today the helper performs replacement but does not give callers a first-class remap payload to continue from (`src/ui/tools/powerpointOpenXml.ts:1457`).

#### 2. Promote XML fallback into a reusable remap API

**File**: `src/ui/tools/powerpointShapeTarget.ts`

**Changes**: Add a resolver that accepts stale Office ids or XML ids and returns the current Office `shape.id`, `shapeIndex`, and XML id when available.

```ts
export interface ResolvedShapeIdentity {
  officeShapeId: string;
  xmlShapeId?: string;
  shapeIndex: number;
}

export async function remapShapeIdentityAfterRoundTrip(...) {
  // direct Office lookup first
  // XML fallback second
  // normalized identity result out
}
```

**Why**: The current XML fallback is the right foundation, but it is still framed as a lookup escape hatch instead of the normal post-round-trip identity strategy (`src/ui/tools/powerpointShapeTarget.ts:10`).

#### 3. Update Open XML-backed tools to return refreshed targeting info

**Files**:
- `src/ui/tools/setSlideTransition.ts`
- `src/ui/tools/addSlideAnimation.ts`
- `src/ui/tools/setSlideNotes.ts`
- `src/ui/tools/clearSlideAnimations.ts`

**Changes**: Have each tool consume the new round-trip result and include refreshed `slideIndex`, `slideId`, and affected `shapeId` values in the tool result payload.

```ts
return {
  resultType: "success",
  textResultForLlm: `Updated transition on slide ${result.finalSlideIndex + 1}.`,
  toolTelemetry: {
    originalSlideId: result.originalSlideId,
    replacementSlideId: result.replacementSlideId,
    finalSlideIndex: result.finalSlideIndex,
  },
};
```

**Why**: Today the tools mostly rely on human/model refresh hints after a failure; the result should carry fresh targeting context proactively.

#### 4. Add regression coverage for stale-object recovery

**Files**:
- `src/ui/tools/powerpointOpenXml.test.ts`
- `src/ui/tools/powerpointShapeTarget.test.ts`
- `src/ui/tools/addSlideAnimation.test.ts`
- `src/ui/tools/setSlideTransition.test.ts`
- `src/ui/tools/manageSlideShapes.test.ts`

**Changes**: Add tests for replacement metadata, shape remap after replacement, repeated transition/animation calls, and shape operations that succeed using returned remap data.

```ts
it("returns replacement slide metadata after round-trip", async () => {
  const result = await replaceSlideWithMutatedOpenXml(context, 1, mutate);
  expect(result.finalSlideIndex).toBe(1);
  expect(result.replacementSlideId).not.toBe(result.originalSlideId);
});
```

**Why**: The failure mode is already known and recurring; this needs dedicated regression tests, not only happy-path tests.

## Phase 2: Add First-Class Native Tools for Images, Charts, Tables, and Diagram Layouts

### Overview

This phase adds explicit PowerPoint-native content tools for the common business artifacts your users actually need: project timelines, work-estimate tables, charts, images, and a small set of opinionated diagram layouts.

### Why This Phase First

Once mutation reliability is fixed, the highest-value feature gap versus other polished presentation assistants is the lack of first-class native visual content tools. This phase unlocks visible user value before spending effort on richer UI orchestration.

### Changes Required

#### 1. Add native PowerPoint image tool

**Files**:

## Phase 3: Build Selection-Aware and Current-Slide-Aware Commands in the UI

### Overview

This phase makes the add-in feel like it is acting on the user's current PowerPoint context instead of forcing them or the model to always work from explicit slide indices and ids.

### Why This Phase First

After the native tools exist, the next bottleneck is UX. Users need to select shapes or slides, describe a change, and trust the add-in to target the right thing. This creates the bridge between low-level tools and a more Copilot-like workflow.

### Changes Required

#### 1. Add a PowerPoint context snapshot model

**Files**:
- `src/ui/lib/office-tool-bridge.ts`
- `src/ui/App.tsx`
- `src/ui/tools/getPresentationStructure.ts`

**Changes**: Add a lightweight PowerPoint context fetch path that periodically reads selected slide ids, selected shape ids, and the active slide from the host, then stores it in React state.

```ts
type PowerPointContextSnapshot = {
  selectedSlideIds: string[];
  selectedShapeIds: string[];
  activeSlideId?: string;
  activeSlideIndex?: number;
};

const [pptContext, setPptContext] = useState<PowerPointContextSnapshot | null>(null);
```

**Why**: The UI currently has no host context model even though PowerPoint selection data is already available (`src/ui/App.tsx:254`, `src/ui/tools/getPresentationStructure.ts:53`).

#### 2. Add selection-aware composer and quick actions in the task pane

**Files**:
- `src/ui/components/ChatInput.tsx`
- `src/ui/components/HeaderBar.tsx`
- `src/ui/components/MessageList.tsx`
- `src/ui/App.tsx`

**Changes**: Show current targeting context near the composer and add small buttons such as `Edit selection`, `Use current slide`, `Insert timeline`, and `Insert estimate table`.

```tsx
<SelectionContextBar
  host="powerpoint"
  activeSlideLabel="Slide 5"
  selectedShapesLabel="2 shapes selected"
  onUseSelection={...}
```

**Why**: This delivers the user workflow you requested even if true inline canvas UI is not feasible.

#### 3. Add ribbon/contextual-tab integration with task pane fallback

**Files**:

**Changes**: Add a minimal PowerPoint-specific command group or contextual-tab integration where supported that deep-links the task pane to context-aware flows such as `Edit selection` and `Insert timeline`.


**Why**: Microsoft's current add-in model supports ribbon commands and custom contextual tabs with task pane fallback; that gives discoverability without depending on unsupported inline slide UI.

#### 4. Add context-aware tool argument resolution

## Phase 4: Add Template-Aware Generation Primitives Using Layouts, Placeholders, and Theme Data

### Overview

This phase turns current deck structure data into a generation substrate so the add-in can create or update slides using the active template instead of only freeform shape placement.

### Why This Phase First

Template-aware generation becomes much more credible once the add-in can trust identities, create native business content, and target the current user context. Doing it earlier would increase ambiguity and reduce testability.

### Changes Required

#### 1. Add structured template metadata output

**Files**:
- `src/ui/tools/getPresentationStructure.ts`
- `src/ui/tools/powerpointShared.ts`

**Changes**: Add a structured output mode that returns machine-usable masters, layouts, placeholders, and theme tokens, while preserving the current readable summary mode.

```ts
return {
  slideCount,
  slideDimensions,
  selectedSlideIds,
  selectedShapeIds,
  masters: [
    {
      id,
      name,
      themeColors,
      layouts: [{ id, name, type, placeholders }],
    },
  ],
};
```

**Why**: Current structure output is rich but text-only, making it harder for tools and prompt builders to consume deterministically (`src/ui/tools/getPresentationStructure.ts:78`).

#### 2. Add a template-aware slide creation primitive

**Files**:
- `src/ui/tools/createSlideFromTemplate.ts` (new)
- `src/ui/tools/manageSlide.ts`
- `src/shared/office-tool-registry.json`

**Changes**: Add a tool that creates a slide from a specified master/layout and fills named/typed placeholders with supplied content.

```ts
layoutId: string;
bindings: Array<{
  placeholderType?: string;
  placeholderName?: string;
  text?: string;
  imageUrl?: string;
  tableData?: string[][];
}>;
```

**Why**: `manage_slide` can already create a slide with a layout (`src/ui/tools/manageSlide.ts:96`), but there is no first-class primitive that binds business content into that layout.

#### 3. Add template-aware business layouts

**Files**:
- `src/ui/tools/insertBusinessLayout.ts`
- `src/ui/tools/createSlideFromTemplate.ts`
- `src/ui/tools/manageSlideShapes.ts`

**Changes**: Teach the business layout tool to prefer existing template layouts and theme colors before falling back to generic placement.

```ts
if (template.hasLayout("Timeline")) {
  return createSlideFromTemplate({ layoutId: template.timelineLayoutId, bindings });
}

return insertBusinessLayoutFallback({ layoutType: "timeline", themeColors });
```

**Why**: For offers and branded proposals, using the deck's real layouts matters more than perfect generality.

#### 4. Add generation guidance and verification hooks

**Files**:
- `.opencode/agents/powerpoint.md`
- `TOOLS_CATALOG.md`
- prompt-building code path if system message composition is host-specific

**Changes**: Update PowerPoint guidance so the agent prefers whatever template-aware primitives exist at that time over raw `add_slide_from_code` whenever the deck exposes usable layouts/placeholders.

```md
- Prefer future template-aware slide-generation primitives when a matching layout exists.
- Use low-level slide editing or code fallback only when no suitable template-aware path is available.
```

**Why**: The existing guidance already nudges the model away from code-first slide creation; phase 4 should make that behavior concrete and reliable.

## Testing Strategy

### Unit Tests

- [ ] `powerpointOpenXml` replacement metadata: verifies old/new slide ids and final index are returned.
- [ ] `powerpointShapeTarget` remap logic: verifies stale Office ids can resolve through XML fallback.
- [ ] `setSlideTransition` / `addSlideAnimation`: verify tool results include refreshed targeting metadata.
- [ ] New registry entries: verify PowerPoint tool registry stays aligned with handlers and generated wrappers.
- [ ] Template metadata serializer: verifies layouts/placeholders/theme colors are emitted in structured form.

### Integration Tests

- [ ] Repeated round-trip mutation flow: transition -> animation -> notes on the same slide without stale-object failure.
- [ ] Native content flow: create timeline, estimate table, and image using first-class PowerPoint tools.
- [ ] Context-aware UI flow: selected shape -> task pane edit -> target resolved correctly.
- [ ] Template-aware generation flow: select a branded deck, create a slide from layout bindings, and inspect output with the current slide-reading/structure tools.

## Risks and Mitigations

| Risk | Mitigation |
| --- | --- |
| Open XML replacement still causes host-specific identity churn | Keep one central round-trip helper, return replacement metadata, and add regression tests around repeated mutations. |
| Native PowerPoint APIs may not support every desired object type uniformly across hosts | Gate by requirement set, provide explicit fallback behavior, and keep `add_slide_from_code` as a controlled escape hatch. |
| Selection-aware UX may drift out of sync with actual PowerPoint selection | Use explicit context refresh before execution, avoid relying on preview-only events, and show visible context labels in the task pane. |
| Contextual tabs may not be supported on all builds/platforms | Implement task-pane-first UX with ribbon/contextual enhancements as optional discoverability layers and documented fallback. |
| Template-aware generation may choose weak or wrong layouts | Start with explicit layout selection + typed placeholder bindings, then add heuristics only after manual verification data. |
| Diagram scope could balloon into generic SmartArt ambitions | Keep phase 2 limited to opinionated business layouts: timeline, phase plan, process flow, comparison, estimate summary. |
